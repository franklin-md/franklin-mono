Based on the vendored Pi sources here (`@mariozechner/pi-agent-core` and `pi-agents`), my read is:

1. Session lifecycle hooks should live elsewhere, not on `SessionAPI`.
Franklin’s session layer is deliberately runtime-only: `createSessionSystem` injects `child`, `fork`, and `removeSelf` onto `runtime.session`, and explicitly does not expose session behavior on the registration API ([system.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/sessions/system.ts:10), [system.test.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/sessions/__tests__/system.test.ts:97)). Lifecycle side effects already compose outside that surface: `SessionCollection` emits add/remove events, and persistence is implemented by subscribing to those events in `PersistedSessionCollection` ([collection.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/sessions/runtime/collection.ts:17), [persisted-session-collection.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/agent/src/agent/session/persisted-session-collection.ts:12)). Pi follows the same split: extensions listen to top-level `session_start`, `session_switch`, `session_fork`, `session_tree`, and `session_shutdown` events via `pi.on(...)`, while session identity/file metadata comes from `ctx.sessionManager` ([pi-agents/index.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/pi-agents/extensions/agent/index.ts:561), [session-events.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/pi-agents/extensions/agent/session-events.ts:45)). So: keep `SessionAPI` imperative and local to a runtime; put lifecycle observation on a manager/collection/app event surface parallel to `CoreAPI.on(...)`, not inside `SessionAPI`.

2. The minimal history primitives are very small.
At the protocol level, Mini-ACP only needs `history = { systemPrompt, messages }`, plus whole-field `setContext` patching; `messages` is replaced wholesale when supplied, not edited via rich history ops ([mini-acp/README.md](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/README.md:96), [context.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/src/types/context.ts:34)). The tracker only performs two mutations: apply a partial context patch and append a message ([ctx-tracker.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/src/protocol/ctx-tracker.ts:41)). Tracking also shows what actually matters in the log: user messages, completed assistant updates, assistant tool-call content, and tool-result messages; chunks are not persisted as history ([tracking.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/src/protocol/tracking.ts:18), [core/runtime.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/core/runtime.ts:44)). Franklin’s persisted core state is even narrower: it stores only `messages` and `llmConfig`; `systemPrompt` is recomputed by the system-prompt decorator rather than treated as durable core state ([core/state.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/core/state.ts:3), [tracker decorator](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/core/compile/decorators/tracker.ts:19), [system-prompt.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/core/compile/decorators/system-prompt.ts:35)). Pi’s own mutators line up with this: `setSystemPrompt`, `replaceMessages`, `appendMessage`, `clearMessages` ([pi-agent-core/agent.js](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/@mariozechner/pi-agent-core/dist/agent.js:132), [agent.js](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/@mariozechner/pi-agent-core/dist/agent.js:216)). So the real minimum is: ordered canonical messages, a system-prompt value, and snapshot/replace/append/clear semantics. No turn objects, chunk history, or built-in summary objects are required.

3. Compaction should compose on top of history, not belong inside it.
Pi already has the right seam: `transformContext(messages)` runs immediately before `convertToLlm`, so compaction/pruning changes the model-facing view without mutating canonical stored history ([pi-agent-core/agent-loop.js](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/@mariozechner/pi-agent-core/dist/agent-loop.js:140), [types.d.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/@mariozechner/pi-agent-core/dist/types.d.ts:107)). The `pi-agents` workflow executor does the same thing structurally: it keeps full `memory.history`, then derives summarized context only when building delegated task prompts ([executor.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/pi-agents/extensions/agent/executor.ts:518), [executor.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/node_modules/pi-agents/extensions/agent/executor.ts:604)). Franklin has no embedded compaction primitive in history today; its natural seam is a pre-prompt decorator/runtime wrapper, analogous to the system-prompt decorator, or an app-side materialization step before `setContext`. History should remain the source of truth; compaction should be a reversible projection over it. Only make compaction mutate stored history if you are explicitly doing archival/GC, not normal turn execution.
