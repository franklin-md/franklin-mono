Franklin / mini-acp mapping

- Natural fit to reverse RPC: `confirm`, `select`, `input`, `editor`. Mini-ACP’s protocol intentionally exposes one client-side reverse RPC, `toolExecute` ([packages/mini-acp/README.md](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/README.md:31), [src/base/types.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/src/base/types.ts:11)). Franklin extensions already register app-executed tools through [CoreAPI.registerTool](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/core/api/api.ts:59). So Pi-style dialogs map cleanly to “ask the host app, wait for a result” tools. The only caveat is shape: Mini-ACP tool results are text/image only ([src/types/tool.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/src/types/tool.ts:18)), so booleans/choices/text would likely be encoded as JSON-in-text, which Franklin already does in [todoExtension](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/extensions/todo/extension.ts:19).

- Require new Franklin-side primitives, but not new core protocol: most presentation methods. Franklin’s extension model is tools + stream handlers + shared stores, not UI slots ([packages/mini-acp/README.md](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/mini-acp/README.md:32), [packages/extensions/README.md](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/README.md:3)). `StoreAPI` is the obvious bridge for Pi-like UI state ([store/api.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/store/api/api.ts:6), [store/runtime.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/systems/store/runtime.ts:14)); built-ins already use it for status, todos, and conversation state ([status/extension.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/extensions/status/extension.ts:8), [todo/extension.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/extensions/todo/extension.ts:15), [conversation/extension.ts](/Users/afv/conductor/workspaces/franklin/yangon-v2/packages/extensions/src/extensions/conversation/extension.ts:18)). But Franklin has no standard primitives yet for `footer status`, `widget above editor`, `working message`, `hidden thinking label`, `theme`, `composer text`, or `tools expanded`.

- Not realistic without a renderer: `custom`, `setFooter`, `setHeader`, `setEditorComponent`, `onTerminalInput`, and component-backed `setWidget`. Pi passes live renderer objects and keybinding/TUI internals directly; Mini-ACP has no render-tree ABI or renderer object model, only messages, stream events, and tool callbacks. These could exist only as host-specific renderer APIs, not as portable Mini-ACP features.
