**Memo**

The runtime is centered on a single control plane, [`createSubagentExecutor`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/subagent-executor.ts:1160>), wired from the extension bootstrap in [`index.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/index.ts:168>). It normalizes every request into exactly one mode (`single`, `parallel`, or `chain`), expands counted parallel tasks before allocating state, enforces recursion depth, resolves agent discovery/cwd/context, then routes to either the foreground runners or the detached async path.

**Main Subsystems**

- **Child-process spawning and session topology.** Foreground runs go through [`runSync`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/execution.ts:403>) and ultimately `spawn()` a Pi child after assembling args/env in [`buildPiArgs`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/pi-args.ts:41>). Async runs are different: [`executeAsyncSingle/Chain`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/async-execution.ts:146>) serialize fully-resolved steps, then launch detached `subagent-runner.ts` under `jiti`; the runner replays execution and writes status/output/result files ([`subagent-runner.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/subagent-runner.ts:731>)). Pi CLI resolution is cross-platform and prefers `node <cli-script>` when possible, falling back to `pi` ([`pi-spawn.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/pi-spawn.ts:72>)).

- **Prompt and arg assembly.** `buildPiArgs` encodes session mode, model, thinking suffix, tools, extensions, system prompt mode, MCP direct tools, long-task spill-to-file, and inheritance flags. A key nuance is that it always injects the prompt-runtime extension so the child can rewrite its final visible prompt in `before_agent_start` ([`subagent-prompt-runtime.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/subagent-prompt-runtime.ts:55>)). That runtime strips inherited `Project Context` and discovered skills blocks while preserving explicitly injected skill content ([tests](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/test/unit/subagent-prompt-runtime.test.ts:33>)). Chain/parallel runs also build task-local instructions like `[Read from:]`, `[Write to:]`, progress paths, and previous-step summaries in [`settings.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/settings.ts:227>).

- **Chain, parallel, and background orchestration.** Foreground chains run through [`executeChain`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/chain-execution.ts:273>): templates resolve `{task}`, `{previous}`, and `{chain_dir}`; sequential steps pass previous output forward; parallel steps fan out via `mapConcurrent`, then aggregate outputs with stable headers for downstream steps ([`parallel-utils.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/parallel-utils.ts:79>)). Clarify UI can edit templates/behavior before launch; for chain it can request background mode explicitly, while top-level async is suppressed when clarify is active until that decision is made. Worktree-backed parallelism is a first-class branch with explicit cwd conflict checks.

- **Artifacts and outputs.** Artifacts are named per run/agent/index and can include input, output, JSONL, and metadata ([`artifacts.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/artifacts.ts:14>)). Single-output handling is careful: it injects an output instruction, snapshots the target file before execution, and after execution prefers agent-written file contents if the file changed; otherwise it persists the assistant text itself ([`single-output.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/single-output.ts:19>)). Tests explicitly pin `do not overwrite a real agent-written file with the receipt` ([`single-execution.test.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/test/integration/single-execution.test.ts:436>)).

- **Observability and live rendering.** Foreground runs stream JSON events into live progress snapshots inside `runSingleAttempt`; async runs expose `status.json`, `events.jsonl`, and `output-*.log`. The UI layer has two channels: an async widget/poller ([`async-job-tracker.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/async-job-tracker.ts:10>), [`render.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/render.ts:148>)) and result renderers for single/parallel/chain with live tool, activity, artifact, and fork badges ([`render.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/render.ts:207>)). Slash-command live rendering uses placeholder snapshots that are incrementally replaced by progress updates in [`slash-live-state.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/slash-live-state.ts:58>).

- **Completion/failure/cancellation plumbing.** Foreground failure is not just process exit code. `detectSubagentError` can override exit `0` when the child emitted a hidden tool failure, and fallback-model retries only happen for retryable provider/model failures, not normal task failures ([tests](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/test/integration/error-handling.test.ts:146>), [`single-execution.test.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/test/integration/single-execution.test.ts:261>)). Normal abort sends `SIGTERM` then `SIGKILL`; intercom-enabled runs can instead `detach` and return success while the child continues ([`execution.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/execution.ts:331>), [`single-execution.test.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/test/integration/single-execution.test.ts:550>)). Async completion is file-driven: [`createResultWatcher`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/result-watcher.ts:15>) watches result JSONs, filters by session/cwd, dedupes by TTL, emits `subagent:complete`, then deletes the file.

**Critical Invariants**

- Exactly one execution mode must be selected.
- Count expansion happens before fork-session allocation and max-task validation.
- Fork mode requires a persisted parent session and current leaf; it creates one branched session per execution index and never silently downgrades to fresh context ([`fork-context.ts`](</Users/afv/conductor/workspaces/franklin/yangon-v2/.context/external/pi-subagents/fork-context.ts:21>)).
- `--session` is exclusive with `--session-dir`/`--no-session`.
- The prompt-runtime extension must always be loaded or inheritance flags become ineffective.
- Parallel ordering is deterministic even when execution is concurrent.
- Fail-fast parallel skips are represented distinctly (`exitCode === -1`), not as ordinary failures.
- Worktree isolation forbids per-task cwd divergence inside a shared worktree parallel group.
- Async observability files must exist before the final result file appears.

**Hardest Parts To Preserve In A Port**

- Foreground/async behavioral parity. The async runner duplicates a lot of foreground logic; drift here is the main architectural risk.
- Prompt rewriting timing. It depends on modifying the child’s final prompt, not just composing text in the parent.
- Forked session semantics. Per-index branched sessions are fundamental to chain/parallel isolation.
- Distinguishing abort from detach. Intercom handoff is intentionally not a normal cancellation.
- Output reconciliation. There are four distinct representations: live output log, final textual result, agent-written output file, and archived artifact.
- Hybrid observability. Foreground uses in-process callbacks; async uses filesystem status/events plus UI polling and watcher coalescing.
